# UL Forge v1 — Development Documentation

> **UL Forge** is the complete toolchain for composing, representing, and rendering Universal Language glyphs.  
> It takes UL-Script typed by humans or generated by AI, processes it through a geometry-native intermediate representation (UL-GIR), and outputs visual renderings faithful to UL's geometric semantics.

---

## What UL Forge Is

UL Forge is the bridge between UL's formal theory and its visual, usable reality. The theory defines 5 geometric primitives, 4 sorts, 13 operations, and 5 spatial relationships — all proven mathematically unique and universal. UL Forge turns that theory into software: a parser that reads UL-Script, a data model that represents geometric meaning faithfully (including intersections, cycles, and cross-references), a layout engine that solves the hard placement problem, and renderers that produce output for screens, paper, and machines.

**Three-layer architecture:**

```
┌─────────────────────────────────────────────────┐
│  LAYER 1: UL-Script (keyboard input)            │
│  Unicode-capable text encoding of glyph structure│
│  △{●} →@60 □{●}                                 │
└────────────────────┬────────────────────────────┘
                     │ ul-parse
                     ▼
┌─────────────────────────────────────────────────┐
│  LAYER 2: UL-GIR (geometric IR)                 │
│  Graph-with-tree-spine JSON representation       │
│  Nodes = primitives, Edges = spatial relations   │
└────────────────────┬────────────────────────────┘
                     │ ul-render
                     ▼
┌─────────────────────────────────────────────────┐
│  LAYER 3: Visual Output                          │
│  SVG (primary), TikZ (academic), PDF, Canvas     │
│  Faithful geometric rendering with layout solving│
└─────────────────────────────────────────────────┘
```

## Quick Start for Developers

```bash
# Phase 1 core tools (what gets built first)
cargo build                     # Build parser + renderer + validator
ul-parse input.ul               # UL-Script → UL-GIR (JSON to stdout)
ul-render input.ul.json         # UL-GIR → SVG (to stdout)
ul-validate input.ul.json       # Check structural + sort correctness
echo '△{●}' | ul-parse | ul-render > output.svg

# Phase 2 API
ul-serve --port 8080            # Start REST + WebSocket server
curl -X POST localhost:8080/parse -d '△{●} →@60 □{●}'

# Phase 3 frontends
npm run dev                     # Web editor at localhost:3000
code --install-extension ul-forge  # VS Code extension
pip install ul-forge && jupyter notebook  # Jupyter magic
```

## Relationship to UL Core

UL Forge is the **implementation** of the theory defined in `ul-core/`:

| Theory (ul-core/) | Implementation (UL Forge) |
|--------------------|--------------------------|
| Symbology → 7 atomic symbols | Parser tokenizes them; Renderer draws them |
| Syntax → 11 algebraic operations | Validator checks sort correctness |
| Grammar → construction rules | Parser enforces valid formations |
| Composition Algebra → 5 spatial relationships | GIR encodes them as typed edges; Layout engine resolves them visually |
| Lexicon → 42 canonical definitions | Template library provides fixed renderings |
| Thesaurus → meaning pathways | Future: semantic search over GIR |

UL Forge does not redefine UL. It faithfully implements the existing specification.

## Design Principles

1. **GIR is canonical.** Everything converts to and from UL-GIR. It is the single source of truth.
2. **Geometry is real.** Points are points, lines have direction, angles have measure. The renderer doesn't approximate — a 60° angle renders at 60°.
3. **Dual legibility.** Every artifact is readable by both humans and machines. UL-Script for humans, JSON-GIR for machines, SVG for both.
4. **Graphs, not trees.** The GIR uses a graph-with-tree-spine model because UL glyphs have intersections, cross-references, and self-reference that trees cannot represent.
5. **Sort safety.** The Σ_UL type system (Entity, Relation, Modifier, Assertion) is enforced at every layer — parse time, GIR construction, and transformation.
6. **Honest engineering.** No fake line counts. No premature optimization. Build Level 1 (templates), then Level 2 (constraints), then Level 3 (optimization).
