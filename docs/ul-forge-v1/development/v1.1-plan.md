# UL Forge v1.1 — Hardening & Completion Plan

> Takes UL Forge from a working prototype (v1.0) to a reliable, tested, production-ready tool (v1.1).
> Scope: fix all issues from the v1.0 audit, close spec-to-implementation gaps from Phases 0–3,
> complete the template library, and establish the test/CI infrastructure needed before Phase 4.

---

## Rationale

The v1.0 critique identified a consistent pattern: the architecture is sound, the core pipeline works,
but the project has prototype-level defensive coding, incomplete test coverage, documentation
contradictions, and security gaps that compound into systemic fragility. Adding Phase 4 features
(AI, collaboration, advanced templates) on top of this foundation would multiply the fragility.

**v1.1 is a quality release.** No new features. Every change either fixes an identified defect,
closes a spec gap, or builds infrastructure that Phase 4 depends on.

## S1 — Panic Safety

**Problem:** 20+ `unwrap()` calls in production code across grammar.rs, templates.rs, and svg.rs.
A malformed pest pair, unexpected GIR structure, or edge case in rendering will crash the process.
In WASM, this corrupts the WebAssembly instance with no recovery.

### S1.1: Parser Error Propagation

**File:** `crates/ul-core/src/parser/grammar.rs` (189 lines, 12 `unwrap()` calls)

Every `.next().unwrap()` on a pest pair iterator must become `.next().ok_or(...)`:

- [ ] `parse_glyph()` line ~53: `pair.into_inner().next().unwrap()` → `.ok_or(UlError::Parse { .. })?`
- [ ] `parse_composition()` line ~69-70: `inner.next().unwrap()` (head pair) → error propagation
- [ ] `parse_composition()` line ~76: `inner.next().unwrap()` (term after operator) → error propagation
- [ ] `parse_term()` line ~83: `pair.into_inner().next().unwrap()` → error propagation
- [ ] `parse_mark()` line ~95: `inner.next().unwrap()` (primitive pair) → error propagation
- [ ] `parse_mark()` line ~99: `content_pair.into_inner().next().unwrap()` → error propagation
- [ ] `parse_primitive()` line ~108: `pair.into_inner().next().unwrap()` → error propagation
- [ ] `parse_primitive()` (Enclosure branch): `inner.into_inner().next().unwrap()` → error propagation
- [ ] `parse_primitive()` (AngleMod branch): `.next().unwrap()` + `.parse::<f64>().unwrap()` → both need error handling
- [ ] `parse_arrow()`: `pair.into_inner().next().unwrap()` → error propagation
- [ ] `parse_operator()` (Connection branch): multiple `.next().unwrap()` → error propagation
- [ ] Replace all `unreachable!()` macros with `UlError::Parse` returns (these are reachable if the pest grammar changes)

**Pattern to apply:**
```rust
// Before:
let inner = pair.into_inner().next().unwrap();

// After:
let inner = pair.into_inner().next().ok_or_else(|| UlError::Parse {
    line: 0,
    column: 0,
    message: "expected inner pair in <context>".into(),
})?;
```

**Stretch:** Extract a helper function to reduce boilerplate:
```rust
fn expect_next(pairs: &mut pest::iterators::Pairs<Rule>, context: &str) -> Result<Pair<Rule>, UlError> {
    pairs.next().ok_or_else(|| UlError::Parse {
        line: 0, column: 0,
        message: format!("expected {context}"),
    })
}
```

### S1.2: Renderer Error Propagation

**File:** `crates/ul-core/src/renderer/templates.rs` (385 lines, 5 `unwrap()` calls)

- [ ] All `gir.node(id).unwrap()` → `gir.node(id).ok_or(UlError::Render { .. })?`
  - Requires changing template functions to return `Result<PositionedGlyph, UlError>` instead of bare `PositionedGlyph`
  - Propagate the `Result` up through `match_template()` return type: `Option<PositionedGlyph>` → `Result<Option<PositionedGlyph>, UlError>`
- [ ] Update all callers of `match_template()` to handle the new `Result`

### S1.3: SVG Writer Cleanup

**File:** `crates/ul-core/src/renderer/svg.rs` (364 lines, 18+ `unwrap()` calls)

The `write!`/`writeln!` calls on `String` buffers are technically infallible, but the `unwrap()` pattern
is noisy and sets a bad precedent. Two options:

**Option A (minimal):** Add a comment block at the top of svg.rs explaining that `write!` to `String`
is infallible and the `unwrap()` calls are safe. This documents the decision.

**Option B (clean):** Replace `write!(svg, ...).unwrap()` with a small macro:
```rust
macro_rules! svg_write {
    ($dst:expr, $($arg:tt)*) => {
        write!($dst, $($arg)*).expect("String write is infallible")
    }
}
```
This makes the safety assertion explicit.

- [ ] Choose Option A or B and apply consistently across svg.rs
- [ ] Ensure no actual I/O-based `write!` calls are hidden among the string writes

### S1 Exit Criteria

- [ ] `grep -r "unwrap()" crates/ul-core/src/` returns zero hits in non-test code (or all hits have an explicit safety comment)
- [ ] `cargo test --workspace` still passes with all `unwrap()` replaced
- [ ] Parser returns `Err(UlError::Parse)` instead of panicking on malformed input
- [ ] Renderer returns `Err(UlError::Render)` instead of panicking on malformed GIR

## S2 — Security Hardening

**Problem:** SVG output has XSS vectors (unescaped labels), API server has missing size limits,
permissive CORS, and default 0.0.0.0 bind address.

### S2.1: SVG Label Escaping

**File:** `crates/ul-core/src/renderer/svg.rs`

Labels are interpolated directly into SVG markup without HTML escaping:
- Line ~79: `aria-label="{label}"` — attribute injection
- Line ~81: `<title>{label}</title>` — HTML injection

- [ ] Add `fn escape_svg(s: &str) -> String` utility that replaces `&`, `<`, `>`, `"`, `'`
- [ ] Apply `escape_svg()` to every label interpolation in svg.rs (aria-label, <title>, <text>)
- [ ] Add test: label containing `<script>alert('xss')</script>` produces safe SVG output
- [ ] Add test: label containing `"onload="alert(1)` produces safe SVG output

**Implementation:**
```rust
fn escape_svg(s: &str) -> String {
    s.replace('&', "&amp;")
     .replace('<', "&lt;")
     .replace('>', "&gt;")
     .replace('"', "&quot;")
     .replace('\'', "&#x27;")
}
```

### S5.2: WASM Panic Safety

**File:** `crates/ul-wasm/src/lib.rs` (38 lines)

Even with S1 replacing parser `unwrap()` calls, unexpected panics from Rust code in WASM corrupt
the instance. Add a `catch_unwind` safety net:

- [ ] Wrap each WASM-exported function body in `std::panic::catch_unwind()`
- [ ] Convert caught panics to `JsError` with a descriptive message
- [ ] Add `panic = "abort"` to `[profile.release]` in the WASM crate (smaller binary, since we catch above)

**Implementation:**
```rust
#[wasm_bindgen]
pub fn parse(input: &str) -> Result<JsValue, JsError> {
    std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
        let gir = ul_core::parser::parse(input).map_err(|e| JsError::new(&e.to_string()))?;
        let json = gir.to_json().map_err(|e| JsError::new(&e.to_string()))?;
        Ok(JsValue::from_str(&json))
    }))
    .unwrap_or_else(|_| Err(JsError::new("internal parser panic")))
}
```
